# DNS Tools

Tools for managing, testing, and validating BIND 9 DNS configuration and zone data.

| Tool | Script | Requires |
|------|--------|----------|
| Generate Reverse Zones | `tools/dns/gen-reverse-zones.py` | Python 3.6+ |
| DNS Query Test | `tools/dns/dns-query-test.py` | `dig` (`dnf install bind-utils`) |
| Zone Consistency Check | `tools/dns/zone-consistency-check.py` | `dig` (`dnf install bind-utils`) |

---

## gen-reverse-zones.py

Reads a BIND 9 forward lookup zone file and generates one or more reverse lookup zone files containing the correct PTR records. IPv4 hosts are grouped by `/24` subnet (one zone file per block). IPv6 hosts are grouped by `/64` prefix.

### Syntax

```
python3 tools/dns/gen-reverse-zones.py <zone_file> [OPTIONS]
```

### Options

| Flag | Short | Default | Description |
|------|-------|---------|-------------|
| `zone_file` | _(positional)_ | _(required)_ | Path to the forward lookup zone file to parse |
| `--output-dir` | `-o` | `.` (current dir) | Directory to write the generated reverse zone files |
| `--nameserver` | `-n` | From SOA | Override the primary nameserver FQDN in the generated zones |
| `--email` | `-e` | From SOA | Override the hostmaster email (dot notation, e.g. `hostmaster.lrn.local`) |
| `--ttl` | `-t` | From `$TTL` | Override the default TTL for generated zones |
| `--print-conf` | `-p` | off | Print ready-to-paste `named.conf` zone stanza(s) to stdout |
| `--no-ipv6` | | off | Skip AAAA records — do not generate IPv6 reverse zones |
| `--dry-run` | | off | Print generated zone content to stdout instead of writing files |

### Examples

**Basic — generate reverse zone files in the current directory:**

```bash
python3 tools/dns/gen-reverse-zones.py /var/named/db.lrn.local
```

**Write to `/var/named/` and print named.conf stanzas:**

```bash
python3 tools/dns/gen-reverse-zones.py /var/named/db.lrn.local \
    -o /var/named \
    --print-conf
```

**Preview without writing any files:**

```bash
python3 tools/dns/gen-reverse-zones.py /var/named/db.lrn.local --dry-run
```

**Override nameserver and email (useful when SOA is minimal):**

```bash
python3 tools/dns/gen-reverse-zones.py /var/named/db.lrn.local \
    -o /var/named \
    -n ns1.lrn.local \
    -e hostmaster.lrn.local \
    --print-conf
```

**Set a custom TTL and skip IPv6:**

```bash
python3 tools/dns/gen-reverse-zones.py /var/named/db.lrn.local \
    -o /var/named \
    -t 3600 \
    --no-ipv6
```

### What It Parses

| Zone file element | Used for |
|-------------------|----------|
| `$ORIGIN` | Resolving relative hostnames to FQDNs |
| `$TTL` | Default TTL in generated zone |
| `SOA` record | `mname`, `rname`, refresh/retry/expire/minimum timers |
| `NS` records | Nameserver lines in the reverse zone |
| `A` records | One PTR record per host, grouped by `/24` subnet |
| `AAAA` records | One PTR record per host, grouped by `/64` prefix |

### Output File Naming

| Record type | Subnet | Output filename |
|-------------|--------|-----------------|
| A (IPv4) | `192.168.1.x` | `db.192.168.1` |
| A (IPv4) | `10.0.10.x` | `db.10.0.10` |
| AAAA (IPv6) | `fd00::/64` | `db.<nibble-reversed-prefix>` |

### Sample Generated Zone

For a forward zone with `webserver IN A 192.168.1.10` and `db01 IN A 192.168.1.20`:

```zone
; Reverse lookup zone: 1.168.192.in-addr.arpa.
; Generated by gen-reverse-zones.py on 2026-04-11 15:00:00
;
$TTL 86400
$ORIGIN 1.168.192.in-addr.arpa.

@           IN  SOA  ns1.lrn.local. hostmaster.lrn.local. (
                    2026041101  ; Serial
                    3600        ; Refresh
                    900         ; Retry
                    604800      ; Expire
                    86400 )     ; Negative cache TTL

; Name Servers
            IN  NS  ns1.lrn.local.

; PTR Records
10          IN  PTR  webserver.lrn.local.
20          IN  PTR  db01.lrn.local.
```

### Integrating with named.conf

Use `--print-conf` to get zone stanzas, then:

```bash
# Validate the generated zone
named-checkzone 1.168.192.in-addr.arpa /var/named/db.192.168.1

# Reload BIND
systemctl reload named
```

### Limitations

- `$INCLUDE` directives in the zone file are not followed
- CNAME records are ignored — PTRs are only created for A/AAAA records
- `$GENERATE` directives are not supported
- Classless `/` delegation is not emitted (plain last-octet PTR owner names only)

## dns-query-test.py

Tests DNS queries against one or more configured servers. Supports A, AAAA, PTR, MX, SRV, TXT, NS, CNAME, and SOA record types. When multiple servers are configured, it queries all of them and flags cross-server disagreements.

### Syntax

```
python3 tools/dns/dns-query-test.py [OPTIONS]
```

### Options

| Flag | Default | Description |
|------|---------|-------------|
| `--name NAME` | IPA SRV records | Hostname or IP to query. If not specified, queries IPA SRV records and zone SOA/NS using values from config. |
| `--type TYPE` | `A` | Record type to query: `A`, `AAAA`, `PTR`, `MX`, `SRV`, `TXT`, `NS`, `CNAME`, `SOA` |
| `--server SERVERS` | From `[dns] servers` | Comma-separated DNS server IP(s) to query (overrides config) |
| `--json` | off | Emit JSON output |

### Examples

**Test A record for a specific host:**

```bash
python3 tools/dns/dns-query-test.py --name ipa01.lrn.local --type A
```

**Test PTR for an IP:**

```bash
python3 tools/dns/dns-query-test.py --name 192.168.1.10 --type PTR
```

**Test SRV records for IPA LDAP:**

```bash
python3 tools/dns/dns-query-test.py --name _ldap._tcp.lrn.local --type SRV
```

**Query all default IPA SRV and SOA records against both nameservers:**

```bash
python3 tools/dns/dns-query-test.py
```

**Query a specific server (not from config):**

```bash
python3 tools/dns/dns-query-test.py --name lrn.local --type SOA --server 8.8.8.8
```

**JSON output:**

```bash
python3 tools/dns/dns-query-test.py --name ipa01.lrn.local --json
```

### Default Query Behavior (no `--name`)

When `--name` is not specified, the tool automatically queries:
- `lrn.local SOA` — zone authority check
- `lrn.local NS` — nameserver delegation
- `_ldap._tcp.lrn.local SRV` — IPA LDAP discovery
- `_kerberos._tcp.lrn.local SRV` — Kerberos discovery
- `_kpasswd._tcp.lrn.local SRV` — kpasswd discovery

These defaults are useful for a quick IPA/DNS health check with no arguments.

### Status Values

| Status | Meaning |
|--------|---------|
| `OK` | Query returned an answer |
| `WARN` | No answer (NXDOMAIN) or cross-server mismatch detected |
| `FAIL` | Query timed out |

**Cross-server mismatch detection:** If querying multiple DNS servers returns different answers for the same name/type, all answers are flagged WARN with `[MISMATCH]` appended. This catches replication or configuration divergence between primary and secondary nameservers.

### Dependencies

Requires `dig` from bind-utils:

```bash
dnf install bind-utils
```

## zone-consistency-check.py

Reads every A record from a forward zone file, queries the corresponding PTR record against a live DNS server, and reports whether the PTR matches the forward name. Identifies:

- Missing PTR records (NXDOMAIN)
- Mismatched PTR records (PTR points to a different name)
- Timeout failures

### Syntax

```
python3 tools/dns/zone-consistency-check.py <zone_file> [OPTIONS]
```

### Options

| Flag | Default | Description |
|------|---------|-------------|
| `zone_file` | _(required)_ | Path to the forward zone file to validate |
| `--server SERVER` | First in `[dns] servers` | DNS server to query for PTR lookups |
| `--json` | off | Emit JSON output |

### Examples

**Basic check using config DNS server:**

```bash
python3 tools/dns/zone-consistency-check.py /var/named/db.lrn.local
```

**Check against a specific server:**

```bash
python3 tools/dns/zone-consistency-check.py /var/named/db.lrn.local \
    --server 192.168.1.11
```

**JSON output for reporting:**

```bash
python3 tools/dns/zone-consistency-check.py /var/named/db.lrn.local --json
```

### Sample Output

```
==============================================
  Zone Consistency Check
==============================================

  Zone: /var/named/db.lrn.local  |  DNS server: 192.168.1.10
  A records: 12  |  Matching PTR: 11

  FQDN                      IP             PTR                        Match  Status
  ────────────────────────  ─────────────  ─────────────────────────  ─────  ──────
  ns1.lrn.local.            192.168.1.10   ns1.lrn.local.             yes    OK
  ipa01.lrn.local.          192.168.1.11   ipa01.lrn.local.           yes    OK
  webserver.lrn.local.      192.168.1.20   webserver.lrn.local.       yes    OK
  oldhost.lrn.local.        192.168.1.99   NXDOMAIN                   no     WARN
```

### Status Values

| Status | Meaning |
|--------|---------|
| `OK` | A record has a matching PTR |
| `WARN` | PTR is missing (NXDOMAIN) or points to a different name |
| `FAIL` | PTR query timed out |

### Exit Codes

| Code | Meaning |
|------|---------|
| `0` | All A records have matching PTR records |
| `2` | One or more mismatches or missing PTR records (warnings) |
| `1` | Tool error (zone file unreadable, all queries failed) |

### Workflow: Fixing Mismatches

1. Run the check to identify mismatched or missing PTRs
2. Use `gen-reverse-zones.py` to regenerate the reverse zone from the forward zone
3. Reload BIND and re-run the check to confirm

```bash
# Find mismatches
python3 tools/dns/zone-consistency-check.py /var/named/db.lrn.local

# Regenerate reverse zones
python3 tools/dns/gen-reverse-zones.py /var/named/db.lrn.local \
    -o /var/named --print-conf

# Reload
systemctl reload named

# Verify
python3 tools/dns/zone-consistency-check.py /var/named/db.lrn.local
```
